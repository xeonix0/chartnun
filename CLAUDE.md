## 실수 예방 운영 원칙

- 작업 시작 전에는 문제를 한 줄로 정의하고, 수정 범위를 가능한 한 좁힌다.
- 수정 전에는 대상 파일을 읽고, 관련 심볼/필드의 사용처를 검색한다. 추측으로 이름을 바꾸지 않는다.
- 한 번에 하나의 가설만 검증한다. 여러 원인이 겹치면 먼저 재현 가능한 최소 사례를 만든다.
- 코드 변경 후에는 반드시 실행 가능한 검증을 한다. "완료"라고 말하기 전, 실제 명령 결과를 확인한다.
- 범위 밖 변경이나 리팩토링은 별도 작업으로 분리하고, 필요하면 docs/MASTER.md에 이슈로 남긴다.
- **관찰만 하고 코드로 안 고치면 나중에 실제 버그로 재발한다** — 코단에서 "USDC 정산 심볼 누락"을 운영 숙지 사항으로만 기록해뒀다가 이후 리스크 한도 과소산정 버그로 실제 재발한 사례가 있다. 발견한 이슈는 기록만 하지 말고 코드로 반영하거나, 명시적으로 "언제까지 미룬다"를 문서에 남긴다.

## 핵심 원칙

- **사실 기반 조사 후 수정** — 코드를 읽고 파일:라인으로 원인 특정 후 수정. 추측 기반 수정 금지
- **단일 출처** — 공통 입출력은 `schema.py`, 요약 포맷은 `summarizer/templates.py`. 중복 정의 금지
- **거래소·프로젝트 특이 지식은 adapters 뒤에** — `core/`·`summarizer/`는 `schema.py`의 `ChartInput`/`Candle`만 알아야 하고, 키움 계좌·Bybit 무기한 정산통화 같은 개념이 새어 들어가면 안 됨. 새 소비 프로젝트(비단 등)가 붙을 때 adapter 하나만 추가하면 되는 구조를 유지하는 게 이 프로젝트의 존재 이유
- **의도 보존 우선** — 기존 코드 우회/skip/삭제 전에 왜 그렇게 짰는지 grep으로 전수 조사
- **근본 해결 > 근사치** — 임시 땜빵보다 근본 원인 해결 우선
- **한 번에 하나씩** — 한 번에 여러 원인을 동시에 손대지 말고, 하나의 가설을 먼저 검증한다
- **검증 없는 완료 선언 금지** — 실제 실행/테스트 결과, 그리고 이 프로젝트 특성상 **실제 생성된 요약 문장/차트 이미지**가 있어야만 완료로 말한다
- **수치 없는 요약은 실패작** — "상승 추세입니다" 류 뭉뚱그린 문장은 스펙 위반. 항상 가격·%·배율 중 하나 이상 포함
- **원본 정밀도 유지** — 가격을 표시/요약할 때 임의로 반올림하지 않는다. 저가 알트코인은 소수점 여러 자리가 유효숫자이고, 이게 손절/청산가 판단에 직결되므로 형식(%.2f 등)을 강제로 걸지 않는다 (코단 DASHBOARD_UI_GUIDE.md §4와 동일 이유)
- **실행 검증이 아니면 못 잡는 버그가 있다** — 코단 세션에서 실제로 겪은 사례: 서버 프로세스가 뜨고 HTTP 200을 받는 것만으로는 부족했고(`streamlit.testing.v1.AppTest`로 실제 스크립트 실행까지 확인해야 잡히는 임포트 충돌 버그가 있었음), 소규모 심볼 테스트로는 안 드러나고 정식 유니버스 규모에서만 드러나는 버퍼/스케일 버그도 있었음. 이 프로젝트도 "단위테스트 초록불"과 "실제 여러 프로젝트에서 다양한 심볼·타임프레임으로 돌려본 것"은 다른 검증 수준이라는 것을 기억할 것
- **패키지/파일명 충돌 주의** — 코단에서 `dashboard/app.py`가 루트 `app/` 패키지와 이름이 겹쳐 재귀 임포트 크래시가 난 적이 있음. 이 프로젝트는 3개 프로젝트에 import되므로, 소비 프로젝트 쪽 패키지명(`app`, `strategies` 등)과 이 라이브러리의 모듈명이 겹치지 않는지 새 소비 프로젝트를 붙일 때마다 확인

## 명령어 정의

**`상태 확인`**
docs/MASTER.md(예정)를 읽고 현재 개발 순서(§8 1~5단계) 중 어디까지 완료했는지, 다음 작업을 요약한다. 코드 수정 금지 — 보고 후 대기.

**`이어서`** (또는 `이어서 진행`)
1. `상태 확인`을 먼저 수행한다.
2. §8 우선순위대로 다음 항목을 STOP GATE를 통과하며 진행한다.
3. 완료 시 `docs/MASTER.md` 체크 갱신 + `docs/HISTORY.md` 이력 추가.

**`[대상] 조사해`**
1. 관련 파일을 읽는다 — **수정 금지**
2. 보고: 확인한 파일:라인 / 현재 동작(사실만) / 문제/의심(근거 파일:라인) / 추천 방향
3. 보고 후 대기.

**`[대상] 수정해`**
1. STOP GATE 1 통과
2. 확인 후 수정 진행
3. STOP GATE 2 통과

**`요약 문장 검수`**
1. 지정한 ChartInput(또는 예시 심볼)에 대해 `analyze_and_summarize()`를 실제로 실행한다.
2. 생성된 문장이 스펙 5장 기준(추세/위치/거래량/패턴/종합 5문장, 각 문장 최소 1개 수치)을 만족하는지 대조한다.
3. 미달 항목이 있으면 `summarizer/templates.py` 또는 `build_summary.py` 수정 대상으로 보고 후 대기.

- **미니PC 배포와의 관계**: 이 프로젝트 자체는 상시 서비스가 아니라 라이브러리라 `d:\www\SHARED_MINI_PC_SERVER.md`에 독립 배포 항목이 없다. 2026-07-08, 주단·코단이 실제로 `uv add git+https://github.com/xeonix0/chartnun.git`로 이 라이브러리를 의존성 추가해 연결 완료(§8 5번) — Windows Credential Manager 인증으로 비공개 저장소 clone이 문제없이 동작함을 확인. 미니PC(`unrules-nucbox-g2`)는 `sudo` 비밀번호가 없어 `uv sync`(사용자 계정 권한)로만 의존성을 설치할 수 있는 제약이 있는데(SHARED_MINI_PC_SERVER.md 참고), git 의존성은 `uv sync`만으로 그대로 동작하는 방식이라 이 제약과 충돌하지 않는다. **다만 미니PC는 리눅스이고 인증 수단(Windows Credential Manager)이 이 개발 머신과 다르므로, 실제 미니PC 배포 시점에 git clone 인증(SSH 키 또는 PAT)이 별도로 필요할 수 있음 — 그때 재확인할 것.**
